# Full i18n (KZ, UA, EN-US, EN-GB) — Strategic roadmap

> Status: scaffold  
> Owner: TBD  
> Linked context: `frontend/src/locales/{kz,ua}/`, `frontend/src/lib/i18n.ts`  
> Last revised: 2026-04-23

## 1. Vision

druz9 ships RU/EN today. To win the second-tier CIS markets (Kazakhstan,
Ukraine) and to professionally serve the global English-speaking audience
(EN-US, EN-GB), we expand i18n to **five locales**. The scaffold creates the
locale folders and registration plumbing; **content translation is a content-team
deliverable, not engineering work**, and is explicitly out of scope for the
scaffold sprint.

## 2. User personas + pain points

### 2.1 Kazakh learner (KZ-locale)
- **Pains:** Russian works for now but the localised version signals
  long-term commitment and meets state-level digital-product expectations.
- **Wants:** Native Kazakh UI, calendar/numbers in `kk-KZ` formats.

### 2.2 Ukrainian learner (UA-locale)
- **Pains:** Strong post-2022 preference against RU UI, even when fluent.
- **Wants:** Ukrainian UI, no fall-through to RU strings.

### 2.3 US/UK English speakers
- **Pains:** Today's `en` is generic; confuses date format, currency, idiom.
- **Wants:** EN-US ($1,234.50, mm/dd/yyyy) vs EN-GB (£1,234.50, dd/mm/yyyy)
  separation.

### 2.4 Content team
- **Pains:** No place to put translations; afraid of breaking JSON.
- **Wants:** Stable JSON file structure per namespace, never reshape on
  their behalf.

## 3. Phase 1 — MVP scope (1 sprint, scaffold only)

Smallest shippable: **`kz/` and `ua/` directories created with byte-identical
copies of `ru/` JSON files; `i18n.ts` registers them as available languages
that fall back to `ru`.** UI shows the language picker, but content stays
Russian until the content team ships translations. NO content translation
done by engineering. EN-US / EN-GB deferred to Phase 2 (more involved
because date/number formatting hooks are needed).

- Locale folders populated with stub JSON copies.
- `Lang` type widens to `'ru' | 'en' | 'kz' | 'ua'`.
- `resources` map gains kz/ua keys.
- `detectLang` recognises `kk`, `uk` browser locales.
- Anti-fallback: kz/ua intentionally fall back to `ru` (NOT `en`) because
  these audiences read RU; documented explicitly in the file header.

## 4. Phase 2 — EN-US / EN-GB + ICU formatting (3-4 sprints)

- Split `en` into `en-US` and `en-GB`; introduce `Intl.NumberFormat` and
  `Intl.DateTimeFormat` helpers in `lib/i18n.ts`.
- Adopt ICU MessageFormat (`{count, plural, one {…} other {…}}`) — current
  flat-string namespaces will be migrated namespace-by-namespace.
- Currency display: `Intl.NumberFormat(locale, { style: 'currency' })`.
- Right-to-left readiness audit (no RTL languages planned yet but the work
  prevents debt if Arabic/Hebrew ever ships).

## 5. Phase 3 — Translation operations (continuous)

- Crowdin / Lokalise integration for content team.
- CI gate: missing keys in non-`ru` locales = warning, not fatal (because
  fallback exists).
- Translation memory + glossary so "ELO", "kata", "Sanctum" stay consistent.
- Per-locale screenshot snapshots in CI to catch overflow / truncation.
- LLM-assisted first-pass translation (review by humans), reusing the
  OpenRouter wiring.

## 6. Database schema deltas

**None for Phase 1.** Phase 2 adds `users.locale` extension to allow
`en-US` / `en-GB` strings (today's column is `TEXT NOT NULL DEFAULT 'ru'`,
no constraint, so the wider values fit without ALTER).

Phase 3 may add a `content_translations` table if we localise dynamic
content (kata text, daily prompts) — explicitly out of scope here.

## 7. API contracts

No new RPCs. Existing endpoints already accept `Accept-Language` header
and `users.locale`. Phase 2 will add a per-request `Accept-Language`
override at the auth middleware so server-rendered email/PDF respects it.

## 8. Frontend pages / components

- Settings → Language picker — extend existing toggle from binary to
  4-way dropdown (5-way after Phase 2 splits EN).
- No new routes.

## 9. Pricing model

i18n is a platform feature, not a paid one. Bundled with all tiers.

## 11. Risks + mitigations

| Risk                                                       | Mitigation                                                                |
| ---------------------------------------------------------- | ------------------------------------------------------------------------- |
| Content team ships partial translation → mixed-language UI | Fallback chain `kz → ru`, `ua → ru`, `en-US → en` keeps every key resolved |
| Layout overflow in long German-style words (future)        | Audit snapshots in CI; reserve 30 % horizontal slack in nav components    |
| Hardcoded RU/EN strings sneaking into new code             | ESLint rule (Phase 2): no string literals in JSX text children            |
| Locale-aware bugs in date math (timezone leak)             | Centralise via `lib/dates.ts` wrapper using `Intl` exclusively            |
| Politically-sensitive translations (UA in particular)      | Native UA reviewer required; never auto-publish LLM-translated UA strings |
